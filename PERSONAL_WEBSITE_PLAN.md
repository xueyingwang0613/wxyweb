# 王雪莹个人网站规划文档

> 版本：V1.0  
> 网站类型：个人 AI 实践档案馆 / AI 产品作品集 / Vibe Coding 实验室 / 轻量合作入口  
> 核心气质：真实、结构化、持续实验、克制表达

---

## 1. 网站整体定位

### 1.1 一句话定位

**一个记录 AI 如何进入真实工作流的个人实践档案馆。**

这里既展示王雪莹作为 AI 产品经理参与过的产品与业务问题，也持续收录她用 Vibe Coding、Skill 和 Workflow 做出的实验、方法与复盘。

### 1.2 首页主标题建议

推荐方案：

> **把 AI 放进真实工作流里。**

副标题：

> 我是王雪莹，一名 AI 产品经理。这里记录我做过的 AI 产品、B 端系统和营销 Agent，也保存一些正在生长的 Vibe Coding 实验与工作流方法。

备选主标题：

- AI 产品实践，也包括还没长成产品的实验。
- 记录我如何把 AI 从想法带进工作流。
- 在产品、业务与 AI 之间，持续做一些真实的小实验。

### 1.3 网站角色

网站同时承担四个角色，但呈现上以“实践档案”为主：

1. **个人实践档案馆**：按时间和主题记录职业经历、项目与思考。
2. **AI 产品作品集**：说明问题、职责、产品机制与反思，不堆砌结果数字。
3. **Vibe Coding 实验室**：展示小项目、原型、过程与下一步，而不只展示完成品。
4. **轻量合作入口**：让合适的人知道可以一起做什么，并提供自然的联系路径。

### 1.4 核心表达

访问者离开网站前，最好能记住三件事：

- 王雪莹是一名有 B 端、营销 AI、客服与 Agent 经验的 AI 产品经理。
- 她关心的不是 AI 概念，而是 AI 如何嵌入真实任务、角色和工作流。
- 她既做相对完整的产品，也通过 Vibe Coding、Skill 和 Workflow 持续动手实验。

### 1.5 主要访问者

- 想了解经历与能力的招聘方、业务负责人或合作伙伴。
- 正在寻找企业 AI 落地、营销 Agent 或工作流设计支持的团队。
- 对 AI 产品、Vibe Coding、Skill / Workflow 感兴趣的同行。
- 因内容、项目或共同兴趣而来，希望进一步交流的人。

### 1.6 明确不做什么

- 不做强转化、强承诺的个人销售页。
- 不用空泛形容词替代项目事实。
- 不包装未完成项目，不虚构业务结果或数据。
- 不把所有内容都堆在首页；首页负责建立认识，详情页负责提供证据。
- 不把 Skill 写成“超长 Prompt 合集”，而是展示可复用的任务方法。

### 1.7 内容原则

- **事实优先**：已完成、进行中、想法阶段明确标注。
- **过程可见**：写清问题、判断、取舍、协作与反思。
- **轻成果叙事**：有可靠数据时再写数据，没有则描述验证方式和当前进展。
- **持续更新**：项目可以有更新日志，Notes 可以短小，不要求每篇都是完整文章。
- **产品经理视角**：每个案例都回答“为谁、在什么场景、解决什么、如何进入流程”。

## 2. 网站导航设计

### 2.1 顶部导航

建议控制为 7 个顶层入口：

| 导航名称 | 路径 | 内容说明 | 二级入口 |
|---|---|---|---|
| 首页 | `/` | 定位、精选实践、最近动态 | — |
| 关于我 | `/about` | 个人介绍、工作方式、关注方向 | 经历 `/experience` |
| AI 产品 | `/ai-products` | AI、Agent、B 端产品案例 | — |
| Vibe Lab | `/vibe-lab` | Vibe Coding 项目与实验 | — |
| Skill & Workflow | `/workflows` | 可复用工作流与方法 | Notes `/notes` |
| 一起合作 | `/collaboration` | 可合作方向与适配方式 | — |
| 联系我 | `/contact` | 联系渠道与交流提示 | — |

### 2.2 导航交互建议

- 桌面端将“经历”放入“关于我”的下拉菜单，将“Notes”放入“Skill & Workflow”的下拉菜单。
- 移动端展开后显示全部 9 个实际入口：首页、关于我、经历、AI 产品、Vibe Lab、Skill & Workflow、Notes、一起合作、联系我。
- 当前页面使用细线、圆点或文字颜色提示，不使用厚重高亮块。
- 右侧可保留一个低强调度的“联系我”描边按钮，但不使用“立即咨询”等销售措辞。
- 页脚重复完整站点地图，并放一句持续更新状态，例如“最近更新：2026.MM”。

## 3. 首页结构设计

首页目标：让访问者在 30 秒内知道“你是谁、做过什么、最近在做什么、是否值得继续点开”。

### 3.1 首屏 Hero

- **模块目的**：快速建立身份、关注的问题和网站气质。
- **推荐内容**：姓名、职业身份、主标题、副标题、当前状态小标签。
- **文案方向**：
  - Eyebrow：`王雪莹 · AI 产品经理`
  - 主标题：`把 AI 放进真实工作流里。`
  - 描述：`这里记录我做过的 AI 产品、B 端系统与营销 Agent，也保存一些正在生长的 Vibe Coding 实验和工作流方法。`
  - 状态标签：`正在探索：个人效率监工 AI / 内容工作流 / 交互小实验`
- **按钮 / 链接**：主按钮“查看 AI 产品实践”；次按钮“逛逛 Vibe Lab”。下方用文字链接“了解我”。
- **视觉建议**：左侧文字，右侧为抽象“工作流节点”或实验桌视觉；避免商务肖像占据大半屏。

### 3.2 快速认识我 / Profile Snapshot

- **模块目的**：用短信息回答“你是谁”。
- **推荐内容**：AI 产品经理、B 端产品经验、重点领域、做事取向。
- **文案方向**：`我在 B 端产品、CRM / ERP、营销 AI、智能客服和 Agent 场景中工作。比起讨论 AI 能做什么，我更关心它如何接住一段具体任务，并成为工作流的一部分。`
- **按钮 / 链接**：“关于我”与“查看经历”。
- **视觉建议**：3—4 个关键词卡片，如 `B 端系统`、`营销 Agent`、`RAG / 客服`、`Workflow`。

### 3.3 精选 AI 产品实践

- **模块目的**：用项目证据承接首屏定位。
- **推荐内容**：首期精选 3 个不同类型案例，例如 AI 营销 Agent、AI 稿件审核、智能客服 / RAG；具体案例以可公开材料为准。
- **文案方向**：不只说“做了什么”，重点写“它进入了哪一段业务流程”。
- **按钮 / 链接**：每张卡片“查看案例”；模块末尾“全部 AI 产品实践”。
- **视觉建议**：横向大卡片；标签区分 `Agent`、`审核`、`RAG`、`B 端系统` 等。

### 3.4 Vibe Coding Lab

- **模块目的**：显示动手能力、好奇心与项目进行时。
- **推荐内容**：
  - 可触摸的云
  - 个人效率监工 AI
  - 小红书内容工作流
- **文案方向**：`有些想法还不需要先成为完整产品。先把它做出来、摸一摸，再决定它应该长成什么。`
- **按钮 / 链接**：项目卡片进入详情；“进入实验室”。如已有 Demo，再显示“打开 Demo”。
- **视觉建议**：截图或短视频优先；明确展示 `已完成 / 进行中 / 想法阶段`。

### 3.5 Skill & Workflow

- **模块目的**：解释你对 AI 能力沉淀的理解，并展示方法论。
- **推荐内容**：一句立场、工作流示意、2—3 个 Skill 示例。
- **文案方向**：
  > Skill 不是更长的 Prompt，而是一份可以被反复调用、持续修订的工作手册。它把输入、判断、步骤、工具与输出组织成稳定的方法。
- **按钮 / 链接**：“查看工作流档案”。
- **视觉建议**：轻量流程图 `输入 → 判断 → 执行 → 输出 → 复盘`，配 Skill 卡片。

### 3.6 经历切片

- **模块目的**：给项目实践补充职业背景，但不在首页展开完整简历。
- **推荐内容**：按阶段概括做过的产品类型与能力变化；具体公司、职位、日期待补充后再发布。
- **文案方向**：从“企业系统与业务流程”到“AI 产品与 Agent”，强调积累如何彼此连接。
- **按钮 / 链接**：“查看完整经历”。
- **视觉建议**：3—4 个时间节点或能力阶段，不放密集职位表格。

### 3.7 最近记录 / Notes

- **模块目的**：让网站保持更新感，呈现日常观察。
- **推荐内容**：最近 3—5 条短文章、项目日志或方法笔记。
- **文案方向**：可围绕 AI 产品判断、Agent 工作流、Vibe Coding 复盘、Skill 设计。
- **按钮 / 链接**：“查看全部 Notes”。
- **视觉建议**：日期 + 标题 + 1 句摘要的编辑部式列表。

### 3.8 可以一起合作的方向

- **模块目的**：自然说明能力可如何被调用，不制造成交压力。
- **推荐内容**：5 个合作方向的简短标签或小卡片。
- **文案方向**：`如果你也在把 AI 放进真实业务、内容生产或个人工作流，我们也许可以一起把问题梳理清楚，再做一个可用的版本。`
- **按钮 / 链接**：“看看可以怎么合作”；次链接“直接聊聊”。
- **视觉建议**：低对比度卡片，不展示价格、限时或夸张承诺。

### 3.9 页尾 CTA / 当前状态

- **模块目的**：给愿意继续交流的人一个温和出口。
- **推荐内容**：当前开放的交流类型、联系方式、更新状态。
- **文案方向**：`欢迎来聊具体问题、正在做的项目，或者一个还没想清楚的 AI 工作流。`
- **按钮 / 链接**：“联系我”。
- **视觉建议**：留白充足的小型文本区域，而非销售横幅。

## 4. 各页面详细内容结构

## 4.1 About / 关于我

### 页面定位

不是简历复述，而是解释“我是怎样的产品经理、为什么关注这些问题、如何工作”。

### 页面模块

#### A. 页面开场

- **文案方向**：`我是王雪莹，一名 AI 产品经理。我长期和企业流程、业务系统与一线工作场景打交道，现在更关注 AI 如何从一次演示，变成能够被真实使用的工作方式。`
- **展示内容**：姓名、职业身份、个人照片或工作现场照（可选）。
- **视觉形式**：大标题 + 短自述，不做传统“个人简介信息表”。

#### B. 我关注的问题

- **文案方向**：围绕三个问题展开：AI 接住了哪段任务？人和 AI 如何分工？怎样把一次有效尝试沉淀成可复用流程？
- **展示内容**：3 个问题卡片，每张包含一句解释。
- **视觉形式**：问题卡 / 大号编号。

#### C. 我的经验地图

- **文案方向**：将经验连接成图，而非简单罗列关键词。
- **展示内容**：B 端产品、CRM / ERP、营销 AI、智能客服 / RAG、Agent、企业提效、内容工作流。
- **视觉形式**：领域地图或关联标签；中心为“真实工作流”。

#### D. 我的工作方式

- **文案方向**：可以提炼为 `先进入场景 → 拆任务与角色 → 找 AI 的有效位置 → 做原型验证 → 沉淀流程与边界`。
- **展示内容**：方法步骤、每一步常问的问题。
- **视觉形式**：五步横向流程图。

#### E. 现在正在探索

- **文案方向**：以进行时写 Vibe Coding、个人效率监工 AI、内容工作流和 Skill 研究。
- **展示内容**：3 张进行中项目卡。
- **视觉形式**：实验便签 / 状态卡。

#### F. 去哪里继续看

- **展示内容**：AI 产品、Vibe Lab、Skill & Workflow 三个入口。
- **视觉形式**：大号文字链接。

## 4.2 Experience / 经历时间线

### 页面定位

呈现职业路径、承担过的问题与能力迁移，不做信息密度过高的在线简历。

### 页面模块

#### A. 经历概览

- **文案方向**：用一段话说明经验从企业系统、业务流程到 AI 产品与 Agent 的连接。
- **展示内容**：经验领域标签。年限、公司、职位等信息仅在确认后填写。
- **视觉形式**：简短导语 + 能力标签。

#### B. 职业时间线

- **文案方向**：每段经历回答四件事：所在阶段、负责领域、典型问题、获得的认识。
- **展示内容**：`时间 / 组织与职位 / 产品与业务范围 / 代表实践 / 能力沉淀`。
- **视觉形式**：纵向时间线；支持点击展开，避免一屏过长。

#### C. 产品领域矩阵

- **文案方向**：展示跨领域经验之间的联系。
- **展示内容**：
  - 企业系统：CRM、ERP、项目管理、业财系统
  - 数据与商品：SPU、数据看板
  - AI 场景：营销 Agent、选号 / 拓号 Agent、稿件审核、智能客服 / RAG
  - 提效场景：企业 AI 工作流
- **视觉形式**：二维矩阵，维度可为“业务领域 × 产品能力”。

#### D. 能力演进

- **文案方向**：从流程建模、复杂系统协作，走向 AI 能力边界、人与 Agent 分工和工作流沉淀。
- **展示内容**：能力阶段与对应案例链接。
- **视觉形式**：分阶段路径图。

#### E. 相关案例

- **展示内容**：与每段经历相关的 AI 产品案例，但需注意脱敏和公开边界。
- **视觉形式**：案例卡片联动。

## 4.3 AI Products / AI 产品实践

### 页面定位

这是核心作品集页面。重点不是功能清单，而是展示如何理解业务问题、设计 AI 介入方式并处理产品边界。

### 页面模块

#### A. 页面导语

- **文案方向**：`我关注 AI 产品是否真正进入任务链：它拿到什么输入，替谁完成哪一步，何时需要人判断，又如何进入下一环。`
- **视觉形式**：大标题 + 一段方法声明。

#### B. 筛选与索引

- **展示内容**：按 `Agent`、`营销`、`审核`、`RAG / 客服`、`B 端系统`、`数据产品`、`企业提效` 分类。
- **视觉形式**：标签筛选；默认显示全部。

#### C. 项目案例列表

- **展示内容**：
  - AI 营销 Agent
  - AI 选号 / 拓号 Agent
  - AI 稿件审核
  - 智能客服 / RAG 应用
  - CRM / 项目管理 / 业财系统
  - SPU / 数据看板
  - 企业 AI 提效场景
- **文案方向**：列表卡片展示一句话、角色、能力标签和当前可公开程度。
- **视觉形式**：大卡片网格或上下交错案例列表。

#### D. 项目详情页

- **建议路径**：`/ai-products/[slug]`
- **内容结构**：项目概览 → 背景问题 → 业务流程 → 我的职责 → 产品方案 → AI 能力与边界 → 验证 / 价值 → 反思。
- **视觉形式**：流程图、功能结构图、脱敏界面、关键决策注释。
- **说明**：没有可靠指标时，用“预期价值 / 验证方式 / 当前观察”替代成果数字。

#### E. 跨项目反思

- **文案方向**：总结多类项目中的共性，例如知识质量、评估机制、人机协同、异常兜底与流程衔接。
- **视觉形式**：短文或原则列表，并链接到 Notes。

## 4.4 Vibe Coding Lab / AI 小项目实验室

### 页面定位

展示“正在做”的个人项目空间。允许不完整，但必须诚实标注阶段、当前可体验内容和下一步。

### 页面模块

#### A. 实验室宣言

- **文案方向**：`这里收留一些还没有商业计划、但值得被做出来摸一摸的想法。代码是材料，AI 是搭档，体验本身是判断。`
- **视觉形式**：大标题 + 手写感注释或实验编号。

#### B. 状态筛选

- **展示内容**：`已完成`、`进行中`、`想法阶段`。
- **视觉形式**：状态圆点 + 筛选标签。

#### C. 项目展示

- **可触摸的云**：突出治愈系互动，以及抓云、拍云、戳云等手势交互。
- **个人效率监工 AI**：突出读取与 Codex 的协作记录，生成每日复盘、阶段统计、需求清晰度评分、返工原因与明日建议的设想。
- **小红书内容工作流**：突出选题、标题、正文和封面方向的流程沉淀。
- **视觉形式**：封面图 / Demo 视频 + 状态 + 一句话 + 更新时间。

#### D. 项目详情页

- **建议路径**：`/vibe-lab/[slug]`
- **内容结构**：灵感 → 当前版本 → 核心交互 → AI 协作方式 → 截图 / Demo → 遇到的问题 → 更新日志 → 下一步。
- **视觉形式**：实验日志；版本节点按时间倒序。

#### E. 实验碎片

- **文案方向**：收录未形成项目的小原型、交互尝试与失败记录。
- **视觉形式**：Masonry 小卡片或短日志流。

## 4.5 Skill & Workflow / AI 工作流沉淀

### 页面定位

解释你如何把重复任务从一次性对话变成可复用、可检查、可更新的工作方法。

### 页面模块

#### A. 核心观点

- **文案方向**：
  > Skill 不是更长的 Prompt。它更像一份给 AI 与人共同使用的工作手册：明确什么时候使用、需要什么输入、按什么步骤判断、调用哪些工具，以及怎样检查输出。
- **视觉形式**：观点大字 + 简洁定义卡。

#### B. 从 Prompt 到 Skill

- **展示内容**：
  - Prompt：解决一次对话中的表达问题。
  - Workflow：把多个步骤、角色与工具连起来。
  - Skill：把稳定方法封装为可重复调用、可持续修订的工作手册。
- **视觉形式**：三列对比表或演进图。

#### C. 我的沉淀方法

- **文案方向**：`识别重复场景 → 定义输入输出 → 拆判断点 → 编排工具与步骤 → 设置检查项 → 在真实使用中更新`。
- **视觉形式**：循环流程图，强调“使用反馈”回到“更新”。

#### D. Skill / Workflow 档案

- **可展示方向**：内容创作、会议纪要、周报、需求分析、企业 AI 落地、小红书内容工作流。
- **视觉形式**：模板化卡片；按场景筛选。
- **说明**：尚未完整沉淀的条目标记“构建中”，不把方向当作已完成成果。

#### E. 工作流详情页

- **建议路径**：`/workflows/[slug]`
- **内容结构**：适用场景 → 问题 → 输入 → 流程 → 判断节点 → 工具 → 输出 → 检查方式 → 版本记录。
- **视觉形式**：流程图 + 输入输出样例 + 更新日志。

#### F. 使用与反思

- **文案方向**：记录哪些步骤稳定、哪些仍依赖人工判断、Skill 在什么条件下会失效。
- **视觉形式**：短 Notes 列表。

- **展示内容**：例如“个人效率监工 AI 构建日志”“Skill 设计笔记”。

## 4.7 Collaboration / 业务合作

### 页面定位

不是服务货架，而是帮助双方判断：哪些问题适合一起讨论、可能以什么方式开始。

### 页面模块

#### A. 轻量开场

- **文案方向**：`如果你正在把 AI 引入业务、内容生产或个人工作流，但还不确定问题该怎么拆、从哪里开始，我们可以一起梳理。`
- **视觉形式**：简短文本 + 一张工作桌 / 流程草图式视觉。

#### B. 可以一起合作的方向

- **展示内容**：五类合作方向，使用本规划第 8 节文案。
- **视觉形式**：五张克制的文本卡；不写价格与夸张收益。

#### C. 可能的合作方式

- **文案方向**：`一次问题梳理 / 一段阶段性共创 / 一个小范围原型 / 面向团队的分享与工作坊`。
- **展示内容**：方式仅作参考，具体以问题为起点。
- **视觉形式**：四步或四种形式标签。

#### D. 比较适合的情况

- **文案方向**：已有具体业务场景；愿意提供真实流程与样本；希望先验证再扩大；接受人机协同和边界设计。
- **视觉形式**：检查清单。

#### E. 开始交流

- **文案方向**：`来信时可以简单告诉我：你在做什么、现在卡在哪里、希望先得到什么。即使问题还不完整也没关系。`
- **按钮 / 链接**：“写信给我”或“查看联系方式”。

句式建议：`面向 [用户 / 角色]，在 [场景] 中帮助完成 [任务] 的 [产品类型]。`

- 原来的业务流程是什么？
- 谁在什么环节遇到了问题？

- 负责范围与角色边界。

- 参与了哪些关键判断与交付物。

- 使用 3—5 个任务导向的功能点。
- 不只写页面名称，写用户通过功能完成了什么。

- 例如：信息抽取、生成、分类、审核、检索增强、Agent 编排、知识库等。

#### 业务价值

- 有真实数据：注明统计口径、范围与时间。
- 暂无公开数据：写“希望改善什么、如何验证、目前观察到什么”，不虚构结果。

#### 我的反思

- 哪个判断后来发生了变化？
- 最难的不是技术，而是什么？
- 如果重做一次，会先验证什么？

### 5.3 推荐视觉结构

`封面 / 流程示意 → 一句话 → 角色与标签 → 背景问题 → 方案与 AI 边界 → 价值 / 验证 → 反思`

## 8. 业务合作模块文案

### 8.1 模块总标题与导语

**可以一起合作的方向**

> 我更喜欢从一个具体问题开始：先看真实场景、现有流程和卡点，再判断 AI 是否适合进入，以及应该先做多大的版本。下面是一些我可以参与的方向。

### 8.2 企业 AI 落地咨询

> 如果团队已经在谈 AI，但还不确定该从哪个场景开始，我可以一起梳理角色、任务、数据与流程，识别适合 AI 介入的位置，并把想法收敛成可验证的方案。重点不是“为了 AI 而 AI”，而是找到真实可用的工作环节。

适合聊：场景梳理、机会判断、流程设计、原型范围、评估与迭代方式。

### 8.3 AI 营销 Agent / 小红书获客工具

> 围绕营销和内容获客场景，一起拆解从线索、选题、内容到跟进的工作链路，设计 Agent 或小工具。可以从一个高频、明确的小环节开始，而不是一开始就追求全自动。

适合聊：营销任务拆解、Agent 方案、内容生产链路、小红书获客工具原型。

### 8.4 企业 AI 培训

> 面向真实岗位和任务设计 AI 培训，不只介绍工具功能。可以结合团队日常工作，现场拆解案例、搭建工作流，并讨论哪些判断应该交给 AI、哪些仍需要人负责。

适合聊：AI 产品认知、岗位工作流、Prompt / Skill / Agent、团队实践工作坊。

### 8.5 个人 AI 工具 / 工作流搭建

> 如果你有一套反复发生、却总靠手工维持的个人工作流程，我们可以一起把它说清楚、拆成步骤，再尝试做成一个轻量工具或可复用 Workflow。

适合聊：信息整理、复盘、写作、项目管理、个人知识与效率工具。

### 8.6 小红书内容优化与陪跑

> 围绕选题、标题、正文和封面方向，一起建立更稳定的内容工作流。重点不是批量生成“标准答案”，而是保留个人表达，同时减少重复劳动，并通过实际发布持续调整。

适合聊：内容定位、选题机制、写作流程、AI 辅助方式、阶段性复盘。

### 8.7 合作模块收尾

> 如果你的问题不完全落在这些分类里，也可以直接说说你正在做什么。很多合作，都是从把一个模糊问题聊清楚开始的。

按钮文案：`说说你在做的事`  
次级链接：`查看联系方式`

## 9. 网站整体风格建议

## 9.1 视觉风格

关键词：**数字档案馆、个人实验室、编辑部、轻技术感、带一点柔软。**

- 整体像一本持续更新的数字研究手册，而不是 SaaS 官网。
- 以排版、留白、细线和内容层级为主，不依赖大面积渐变与炫技 3D。
- 项目状态、版本号、更新时间、实验编号可成为稳定的视觉语言。
- “可触摸的云”可以给网站带来少量柔软、治愈的视觉线索，但不要让全站变成童趣风。
- 图片优先使用真实界面、流程草图、项目截帧和实验素材；没有素材时宁可留白。

## 9.2 配色建议

### 推荐方案：纸张白 + 墨色 + 云雾蓝 + 实验绿

| 用途 | 色值建议 | 说明 |
|---|---|---|
| 页面背景 | `#F7F7F2` | 温和的纸张白，降低商业官网感 |
| 主文字 | `#202321` | 接近墨色，阅读舒适 |
| 次级文字 | `#666B68` | 日期、说明、元信息 |
| 分隔线 | `#DDE1DC` | 卡片和流程线 |
| 主强调色 | `#6C86A6` | 云雾蓝，用于链接与重点 |
| 实验状态色 | `#91A98D` | 低饱和绿，用于进行中 |
| 柔和辅助色 | `#D8C7B3` | 暖沙色，用于 Notes 或标签 |

状态色需要同时配文字或图标，不能只依赖颜色表达。

## 9.3 字体建议

- 中文正文：`PingFang SC`、`Noto Sans SC` 或 `Source Han Sans SC`。
- 中文标题：可继续用无衬线保持理性；Notes 大标题可使用 `Noto Serif SC` 增加编辑感。
- 英文与数字：`Inter` 或 `Geist`，用于时间、版本号、标签和界面元信息。
- 正文建议 16—18px，行高 1.7—1.85；长文内容宽度控制在 680—760px。
- 字体种类最多 2 套，避免“实验感”变成视觉噪声。

## 9.4 页面布局建议

- 桌面端内容最大宽度 1200—1280px，正文阅读区更窄。
- 使用 12 栏栅格；Hero 与项目详情可采用 5:7 或 4:8 的不对称布局。
- 首页每屏只回答一个问题：我是谁 → 做过什么 → 正在做什么 → 如何思考 → 如何联系。
- 案例列表避免等高小卡片堆满页面；优先使用大卡片和可读摘要。
- 详情页固定使用相同内容骨架，降低浏览成本。
- 移动端把筛选标签做成横向滚动，并确保 Demo、截图和长流程图可自然缩放。

## 9.5 交互动效建议

- 首屏：工作流节点缓慢漂移或连线轻微流动，幅度克制。
- 卡片悬停：上移 2—4px、边线变深、显示“查看案例”，不做大幅翻转。
- 项目状态：进行中可有轻微呼吸圆点；尊重 `prefers-reduced-motion`。
- 流程图：滚动进入时逐节点出现，帮助理解顺序。
- “可触摸的云”可在 Vibe Lab 局部提供鼠标 / 手势响应彩蛋，不扩散到全站。
- 页面切换使用 150—250ms 淡入；避免影响阅读和加载速度。

## 9.6 技术栈建议

### 推荐 V1

- **框架**：Next.js（App Router）+ TypeScript。
- **样式**：Tailwind CSS 或 CSS Modules；如果重视独特视觉，可用 CSS Variables 建立轻量设计系统。
- **内容管理**：本地 MDX。项目、Workflow、Notes 都以内容文件维护，便于版本管理与后续迁移。
- **交互**：Framer Motion 仅用于少量过渡；复杂手势实验独立封装。
- **图表 / 流程**：静态 SVG 优先；需要动态关系时再使用 React Flow。
- **图片**：Next/Image + WebP / AVIF；Demo 视频提供封面和懒加载。
- **部署**：Vercel 或 Cloudflare Pages。
- **数据分析**：可选 Plausible 或 Umami，保持轻量、隐私友好。
- **联系表单**：V1 可直接使用邮箱链接；确有需求再接入表单服务和反垃圾机制。

### 为什么适合

- 内容型页面与交互实验可以共存在一个项目中。
- MDX 适合持续增加案例、日志和 Notes，不需要先搭复杂后台。
- 项目状态、标签、更新时间等字段可以通过统一 Frontmatter 管理。
- 后续可逐步加入搜索、RSS、案例筛选和 Demo，不必在 V1 一次完成。

### V1 不建议先做

- 复杂 CMS、自建后台和账号系统。
- 全站 WebGL 或重型 3D。
- 没有内容基础时就做全文搜索。
- 自动轮播、弹窗获客、在线客服浮窗。

### 10.5 内容关系

## 结语

这个网站不需要一次性证明所有事情。它更适合像一个真实的实验室：完成的项目留下结构化案例，进行中的想法留下版本和过程，反复发生的工作被沉淀成 Skill，新的判断则进入 Notes。

网站真正的长期价值，不是“看起来内容很多”，而是让每一次实践都有地方继续生长。
